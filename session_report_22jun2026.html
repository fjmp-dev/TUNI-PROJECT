Error 9000 is the root cause of network instability. When the robot enters Play mode, the safety PLC and Emergency Stop heartbeat failures cause the network stack to collapse. When re-paused, services become briefly available again before collapsing. The robot cannot sustain network connectivity while in Running mode.

REST API Module in UI

When the MiR is stable (Pause mode), the web UI displays via polling every 4 seconds:

  • Battery percentage with color-coded bar (green >40%, orange 20-40%, red <20%)
  • Position (x, y, orientation in degrees)
  • Velocity (linear m/s, angular rad/s)
  • Mode, Mission text, Error count
  • 60-second cache with stale indicator when data is old

MiR Bridge Watchdog

The mir_raw.py bridge (written by Eemil) does not handle disconnection: when the MiR rosbridge drops, the ROS2 node keeps running but receives no data. A watchdog was added (scripts/mir_watchdog.sh) that kills the bridge if no stdout activity is detected for 90 seconds. The entrypoint loop automatically restarts it. Zero modifications to Eemil's code.

Recommendation for Tutor

  • Full computer restart of the MiR (shutdown from tablet, wait 1 minute for capacitor discharge, power on)
  • Verify cables for the SICK Safety PLC and Emergency Stop
  • If error 9000 persists, contact MiR support (possible hardware issue)
  • Full documentation: docs/mir_connectivity_issue.md

4. Bugs Found and Fixed

BugFileSymptomRoot CauseFix
Joint values not appearing in UI joint_server.py (new) UI showed "--" instead of values No HTTP server for /joint_states. Attempt with rosbridge.js failed silently (QoS mismatch). Attempt with docker exec + YAML parsing was slow and fragile. Created joint_server.py: persistent HTTP server in mir_ur_driver :9091, subscribes once to /joint_states (QoS VOLATILE). Backend does simple HTTP proxy. Deployed without container restart via docker cp + docker exec -d.
scaled_joint_trajectory_controller deadlock joint_mover.py Goals accepted but never completed (timeout >15s) The scaled controller internally deadlocks: accepts goals but the result future never resolves. Switched to standard joint_trajectory_controller. The scaled version is inactive. u_start.sh explicitly activates the standard controllers.
joint_mover.py AttributeError crash joint_mover.py:101 Script crashes: 'NoneType' object has no attribute 'status' When rclpy.spin_until_future_complete expires, result_future.result() returns None. Code accessed .status without None check. Added None check: if result is None, return "timeout waiting for result (controller busy or previous goal still running)".
Backend returns 500 with empty message main.py UI shows "joint_mover failed: " (blank after colon) docker exec_run had stderr=False, discarding the Python traceback. stdout was empty because script crashed before printing. Changed to stderr=True. Error message now includes exit code and output.
Robot not moving despite controller reporting SUCCESS ur_start.sh, main.py Controller says "moved from X to Y" but physical robot position unchanged robot_program_running = False. The UR driver reads joints via RTDE but cannot send movement commands without the external control URScript running on the robot. Controller tolerance (0.1 rad) is larger than commanded delta (0.01 rad), so controller thinks the arm is already at the target. Added resend_robot_program service call in ur_start.sh (auto on driver start). Added auto-recovery in backend: if move fails with "goal rejected", resend + reactivate controller, then retry.
Right arm goal rejected ur_start.sh Right elbow move fails with "goal rejected" right_controller_stopper node deactivates the right_joint_trajectory_controller when it detects robot state transitions. Controller left in inactive state. Added reactivation loop in ur_start.sh (3 retries with 3s delay). Backend auto-recovery also reactivates controllers before retrying.
UI Hz label flickering (5 vs 400) index.html "12 joints @ 5Hz" alternated with "12 joints @ 400Hz" Two JavaScript functions (renderJoints via rosbridge, renderJointsFromREST via polling) both updated the same DOM element with different text. Unified to single label: "12 joints @ 400Hz". Removed the parenthetical "(age X.XXs)" that also flickered.
launch_dashboard_client default=false duo_ur_real.launch.py (Eemil's) dashboard_client_node never launched Eemil's launch file sets launch_dashboard_client default to "false". It should be "true" for real hardware operation. Explicitly pass launch_dashboard_client:=true in ur_start.sh. The dashboard client exits with warning on PolyScopeX (expected), but at least the launch attempts it.

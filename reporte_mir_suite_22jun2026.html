1 Integracion del MiR200

El MiR200 es un robot movil que navega por el laboratorio de forma autonoma. Tiene su propio sistema ROS1 interno con 199 topics de telemetria: posicion, velocidad, escaneres laser, mapa, bateria, estado del robot, camaras, sensores. Ademas tiene una API REST en el puerto 80 que nos permite consultar datos basicos sin necesidad de ROS.

El contenedor mir_mir y el bridge de Eemil

Creamos un contenedor Docker nuevo (mir_mir) dedicado exclusivamente a la comunicacion con el MiR. Dentro corre el codigo mir_raw.py que Eemil escribio, sin modificarlo. Lo montamos como volumen para respetar la regla de no tocar proyectos originales.

El bridge hace lo siguiente: se conecta al rosbridge del MiR via WebSocket, descubre automaticamente los 199 topics, y por cada uno crea un publicador equivalente en ROS2. Asi, los datos del MiR aparecen como topics nativos de ROS2 en Kevin.

El problema: el MiR es inestable

El MiR200 pierde la conexion de red con frecuencia, especialmente cuando esta en modo Running. Cuando esto pasa, mir_raw.py se queda "vivo pero sin datos": el proceso sigue ejecutandose pero nunca recibe mas mensajes. No se muere, no se reconecta -- solo se congela. Y como no se muere, el ciclo de reinicio del entrypoint nunca se activa.

Solucion: un watchdog. Escribimos un script (mir_watchdog.sh) que corre en segundo plano y revisa cada 30 segundos si el bridge esta realmente procesando datos. Si pasan 90 segundos sin actividad, el watchdog mata el proceso. El entrypoint del contenedor tiene un while true que lo relanza automaticamente. Cero modificaciones al codigo de Eemil: solo matamos y relanzamos.

El panel del MiR en la pagina web

Cuando el MiR esta en Pause, su API REST responde correctamente. Construimos un proxy en el backend (/api/mir/status) y un panel en la pagina que muestra:

La pagina consulta la API cada 4 segundos. Si el MiR no responde temporalmente, guarda los ultimos datos buenos por 60 segundos con una marca de "datos viejos", para que no se pierda todo de golpe.

Recomendacion para recuperar el MiR

  1. Reinicio fisico completo: desde la tablet, Service > Shutdown, esperar 1 minuto, encender con el boton lateral
  2. Si los errores 9000 persisten, revisar los cables internos del SICK Safety PLC y del boton de emergencia
  3. Si nada funciona, contactar soporte de MiR -- posible dano en el hardware del PLC de seguridad

2 Control de los Brazos UR5e

Los dos brazos UR5e estan montados sobre el torso del MiR200, en posicion invertida: apuntan hacia la espalda del robot, no hacia el frente. Cada brazo tiene su propia IP (192.168.1.102 izquierdo, 192.168.1.103 derecho) y 6 articulaciones. Hoy logramos ver las 12 articulaciones en tiempo real en la pagina web y mover los codos de ambos brazos con botones desde el navegador.

Paso 1: Ver los joints en la pagina

El driver UR de Eemil (duo_ur_real.launch.py) ya publicaba las posiciones de los 12 joints en el topic /joint_states a 400 Hz. El reto era hacer que esos datos llegaran a la pagina web de forma confiable.

Lo que intentamos y no funciono: conectarnos al rosbridge desde JavaScript (fallaba por problemas de calidad de servicio QoS) y ejecutar un proceso nuevo por cada consulta (demasiado lento, casi un segundo por peticion).

Lo que si funciono: crear joint_server.py, un mini-servidor HTTP que vive dentro del mismo contenedor del driver. Se suscribe a /joint_states una sola vez, guarda el ultimo mensaje en memoria, y cuando la pagina pregunta responde con un JSON que contiene los 12 valores. Esto es 10 veces mas rapido que la alternativa y no gasta recursos del sistema.

Ventaja clave: lo desplegamos sin reiniciar el contenedor. Usamos docker cp para copiar el archivo y docker exec -d para lanzarlo en segundo plano. El driver UR siguio corriendo sin interrupcion, los brazos nunca se desconectaron.

Paso 2: Mover los codos desde la pagina

Una vez que veiamos los joints, necesitabamos poder moverlos. Creamos joint_mover.py, un script que recibe tres parametros -- que brazo, que articulacion, que tanto mover -- y le envia la trayectoria al controlador del robot.

En la pagina agregamos 8 botones para controlar los codos. Estan organizados por brazo y por tamano de paso:

BrazoPaso grandePaso chico
Izquierdo-0.1 / +0.1 rad (~6 grados)-0.01 / +0.01 rad (~0.6 grados)

Por seguridad, solo movemos los codos en esta etapa. Las munecas no se tocan. El sistema respeta exactamente el delta que le pedimos.

Figura 4: Los tres obstaculos y como los resolvimos.
ProblemaQue pasabaSolucion
1. Programa detenido El controller reportaba "movimiento exitoso" pero el brazo fisico no se movia ni un milimetro. El script de control (URScript) dentro del robot se habia detenido. El controller pensaba que todo estaba bien porque la tolerancia de posicion (0.1 rad) es mayor que el movimiento comandado (0.01 rad). Agregamos resend_robot_program, un servicio de ROS que reenvia el script de control externo al robot. Se ejecuta automaticamente al iniciar el driver y tambien como recuperacion si un movimiento falla.
2. Controlador apagado (brazo derecho) El brazo derecho rechazaba todos los comandos con "goal rejected". El right_controller_stopper, un sistema de seguridad del UR driver, habia apagado el controlador de trayectoria durante la conexion inicial. Agregamos una reactivacion automatica del controlador con ros2 control switch_controllers. El script de inicio del driver lo intenta 3 veces con pausas. El backend tambien lo intenta como recuperacion.
3. Scaled controller roto Habia un controlador "escalado" que en teoria es mas seguro, pero descubrimos que esta roto: acepta comandos pero jamas los ejecuta. El resultado nunca llega y el sistema se queda esperando para siempre. Desactivamos el scaled controller y usamos el joint_trajectory_controller estandar de ROS2, que funciona perfectamente. El scaled queda apagado hasta que alguien lo arregle.

Recuperacion automatica

Para que la pagina sea confiable sin intervencion manual, el backend ahora detecta cuando un movimiento falla. Si el error es "goal rejected", intenta recuperarse solo: reenvia el programa de control al robot, reactiva el controlador, espera un segundo, y reintenta el movimiento. Si funciona, la pagina responde "OK (recovered)". Si no, muestra el error para que sepas que paso.

Sobre los robots

Los UR5e usan PolyScopeX (software nuevo de Universal Robots). Esto significa que no tienen el servidor de dashboard clasico -- ese solo existe en los e-Series. Para recuperar al robot de un error, hay que hacerlo manualmente desde la tablet. Para control normal, no es problema: el driver funciona en modo headless.

Sobre la arquitectura

Un servidor HTTP persistente dentro del contenedor es mucho mejor que lanzar procesos nuevos por cada consulta. Es 10 veces mas rapido, gasta menos recursos, y se puede desplegar sin reiniciar el contenedor -- critico cuando hay conexiones activas con los brazos que no queremos perder.

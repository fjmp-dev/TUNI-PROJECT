En una frase: el robot está sano por dentro. El problema que lo hace desconectarse es la señal de WiFi, que le llega muy débil. Además encontramos y arreglamos un fallo en nuestro propio programa que empeoraba las cosas.

Regla de oro: el robot y la suite (Kevin) tienen que estar en la misma red: la del Teltonika (192.168.1.x). Si el robot se conecta a la red dd-wrt (192.168.12.x), Kevin no lo alcanza y la suite deja de verlo.

¿Por qué se desconecta? (el WiFi, sin tecnicismos)

El robot se conecta por WiFi en la banda de 5 GHz, que es muy rápida pero de corto alcance. La señal le llega tan floja (−87 dBm) que apenas se sostiene, y cuando se mueve o hay un obstáculo, se cae de la red.

Es como hablar por walkie-talkie estando casi fuera de cobertura: a ratos se entiende, pero en cuanto te alejas un poco, se corta.

Lo vimos en directo: durante el diagnóstico el robot se cayó de la red por completo y tardó más de un minuto sin volver. No fue casualidad — es justo el problema.
La solución: pasar el robot a la banda de 2.4 GHz (más lenta, pero llega mucho más lejos). Misma red Teltonika, solo otra "puerta".
Pendiente: el robot solo ve la red de 5 GHz y no tenemos la contraseña del WiFi → hay que pedírselo a Wael (ver abajo).

¿Qué arreglamos hoy? (el "vigilante" del puente)

La suite tiene un puente que traduce los datos del robot para mostrarlos en la web. Ese puente tiene un vigilante que debe reiniciarlo si se traba.

AntesAhora
El vigilante reiniciaba el puente sano cada 90 s por error (miraba la señal equivocada). Y cada reinicio de más ayudaba a colgar al robot. El vigilante ahora mira los datos reales del robot y le da tiempo a arrancar. Solo reinicia cuando de verdad está trabado.
Probado en vivo: el puente ahora arranca bien, se mantiene estable sin reinicios falsos, y cuando lo "congelamos" a propósito, el vigilante lo detectó y lo levantó solo.

Cómo movemos un brazo (de tu clic al motor)

Cuando mueves un joint desde la web, no enviamos voltajes ni comandos de motor. Enviamos una pequeña trayectoria (dónde está ahora → dónde debe terminar, en X segundos) y el controlador del robot se encarga del resto. Este es el viaje del dato:

Navegadortu clic en el panel UR
→
mir_uiFastAPI: valida + autoriza
→
joint_moverarma la trayectoria
→
controllerros2_control interpola
→
driver URRTDE por Ethernet
→
UR5emueve el motor

En paralelo, joint_server publica la posición actual de los joints (suscrito a /joint_states) — el visor 3D y joint_mover la leen de ahí.

  1. El navegador manda POST /api/ur/move con el cuerpo { arm, joint, delta } (p. ej. right, elbow, +0.05 rad) y tu token de sesión.
  2. mir_ui (FastAPI) valida con Pydantic: brazo y joint de una lista cerrada, y |delta| ≤ 0.5 rad. Aquí está el candado de seguridad: un comando enorme o mal formado nunca llega al brazo.
  3. mir_ui hace un docker exec dentro del contenedor del driver y lanza joint_mover.py right elbow 0.05 — en un hilo aparte, para no congelar la API ni el streaming de joints durante el movimiento.
  4. joint_mover pide la posición actual de los 6 joints a joint_server (HTTP :9091). Si no la consigue, aborta: nunca parte de un «0.0 fantasma» que provocaría un salto brusco.
  5. Calcula el objetivo: copia las 6 posiciones actuales y suma el delta al joint elegido. Le pone una duración (≈ 3 s para 0.1 rad) para que el movimiento sea suave.
  6. Arma un goal de FollowJointTrajectory (acción ROS 2): una mini-trayectoria de 2 puntos — punto 0 = posición actual (t = 0), punto 1 = objetivo (t = duración) — y la envía a /{arm}_joint_trajectory_controller.
  7. El joint_trajectory_controller (ros2_control) acepta el goal e interpola entre los dos puntos, mandando comandos de posición al ritmo de control.
  8. El driver de UR traduce esos comandos a RTDE (Real-Time Data Exchange, por Ethernet, puertos 30001-30004) hacia el controlador del UR5e, que mueve el motor físico.
  9. De vuelta: el controlador responde SUCCEEDED → joint_mover imprime «OK» → la API responde { status: "ok" } → la web te lo confirma, y la nueva posición aparece en /joint_states (visor 3D + lectura en vivo).

Lo que enviamos al brazo, en concreto, es esto — una trayectoria en espacio de joints, no comandos de motor:

# goal: control_msgs/action/FollowJointTrajectory
joint_names: [right_shoulder_pan_joint, ... , right_wrist_3_joint]
points:
  - positions: [ ...posición ACTUAL de los 6 joints... ]   # t = 0 s
  - positions: [ ...igual, pero elbow += 0.05 rad ... ]   # t = 3 s
Si algo falla (goal rechazado, controlador no disponible o timeout — típico si el programa externo del UR se detuvo): la suite llama a resend_robot_program, reactiva el controlador y reintenta una vez sola, en automático.

Cómo fluyen los datos en la MIR Suite

De los robots físicos al navegador, paso a paso. Cámara, brazos UR5e y MiR200.

1. El panorama (paso 0)

Qué piezas hay y cómo se hablan antes de entrar en cada flujo.

Todo corre en Kevin (la Jetson AGX Orin) dentro de contenedores Docker. Tú abres un navegador y la página pide datos; nunca instalas ROS en tu laptop.

  • contenedor mir_ui — sirve la página web y el backend (la API REST en :8080).
  • contenedor mir_camera — habla con la cámara Orbbec por USB.
  • contenedor mir_ur_driver — el driver de los brazos UR5e + rosbridge + joint_server.
  • contenedor mir_mir — el puente que traduce datos del MiR (ROS1 → ROS2).

Dos "idiomas" conviven aquí: ROS 2 (cómo hablan los robots y los nodos entre sí dentro de Kevin) y HTTP/WebSocket (cómo le llega al navegador). Buena parte del trabajo es traducir de uno al otro.

Truco clave: todos los contenedores usan network_mode: host — comparten la red de Kevin. Por eso un contenedor puede hablarle a otro como si fuera localhost, y ROS 2 (que usa DDS) se descubre entre contenedores sin configurar nada. Se hizo así porque el kernel de la Jetson no soporta las redes "bridge" normales de Docker.

Paso a paso:

  • La cámara se conecta por USB a Kevin. El contenedor mir_camera lanza el nodo de Orbbec (ros2 launch orbbec_camera gemini_330_series.launch.py).
  • Ese nodo publica un topic de ROS 2: /camera/color/image_raw/compressed. "Compressed" = la imagen ya viene como JPEG (más liviana para la red).
  • rosbridge (que vive dentro de mir_ur_driver) escucha ese topic y lo expone por WebSocket. Al hacerlo, convierte los bytes del JPEG a base64 (texto), que es lo que un navegador puede meter directo en una imagen.
  • El navegador usa la librería roslib.js para suscribirse por WebSocket y pone el resultado en <img src="data:image/jpeg;base64,…">.

Conversiones: imagen cruda → JPEG (en la cámara) → base64 (en rosbridge). Transporte: USB → DDS (ROS 2) → WebSocket.

  • Los brazos hablan por RTDE (Real-Time Data Exchange, el protocolo propio de Universal Robots, sobre TCP). El driver UR (paquete duo_ur del workspace de Eemil) mantiene esa conexión y publica el topic /joint_states a unos 400–490 Hz.
  • Un topic de ROS 2 no lo puede leer un navegador directo. En vez de abrir el topic cada vez (lento), hicimos joint_server.py: un pequeño servidor HTTP en :9091 que se suscribe una sola vez a /joint_states y siempre tiene a mano el último valor en formato JSON.
  • El backend (/api/ur/joints) le hace de proxy a ese servidor (con un cache de 100 ms para no saturarlo).
  • La página consulta /api/ur/joints cada 250 ms (un único "poller" compartido que alimenta tanto el panel de números como el visor 3D).

Por qué un servidor HTTP y no docker exec por petición: abrir un proceso por cada lectura tardaba ~200 ms; el servidor persistente responde en ~5–10 ms.

  • La página manda un POST /api/ur/move con {arm, joint, delta} (qué brazo, qué articulación, cuánto en radianes).
  • El backend valida con un modelo Pydantic: el brazo debe ser left/right, la articulación una de las 6 válidas, y el delta dentro de un límite seguro. Si algo no cuadra, lo rechaza antes de tocar el robot.
  • Como ejecutar el movimiento toma segundos, el backend lo manda a un hilo aparte (no congela el resto de la API mientras el brazo se mueve).
  • joint_mover.py lee la posición actual, calcula la meta (actual + delta), y la convierte en una acción de ROS 2 (FollowJointTrajectory) que envía al controlador de trayectorias. Ese controlador habla con el hardware por RTDE.

Conversión central: JSON del navegador → action goal de ROS 2 (una trayectoria de 2 puntos: posición actual → meta).

Seguridad: los brazos están montados viendo hacia la espalda del MiR, así que las direcciones +/− pueden ser contrarias a lo esperado. Por eso hay límites en el delta y solo se mueve el codo en pruebas.

Aquí la conversión es a un servicio de ROS 2 (un "llamado": petición → respuesta), no a una acción. El backend traduce {arm, mass, cog} a una llamada ros2 service call … set_payload. El brazo guarda esos valores para compensar la gravedad y detectar colisiones. No hay movimiento.

CoG = centro de gravedad: dónde está concentrado el peso de la herramienta, medido en metros desde la brida de la muñeca.

  • El MiR trae su propia API REST (HTTP en el puerto 80). El backend le hace GET http://192.168.1.13/api/v2.0.0/status y recibe un JSON grande.
  • El backend mapea ese JSON a uno más limpio (battery_percentage → battery_pct, state_text → state, etc.).
  • El MiR responde a ratos; por eso el backend guarda el último valor por 20 segundos. Si el MiR no contesta, sirve el último conocido marcado como stale (viejo), y la UI lo avisa con su antigüedad.
Importante: el MiR solo es estable en Pause. En Play/Running su red se colapsa (errores de seguridad 9000), así que la telemetría solo es fiable con el robot pausado.

El MiR por dentro corre ROS 1 (nosotros usamos ROS 2). El contenedor mir_mir corre mir_raw.py (código de Eemil) que se conecta al rosbridge del MiR por WebSocket, descubre automáticamente sus topics y, por cada uno, crea un publicador equivalente en ROS 2.

Conversiones ROS1 → ROS2 que hace al traducir cada mensaje:

  • Time: secs/nsecs (ROS1) → sec/nanosec (ROS2).
  • Header: se elimina el campo seq (ya no existe en ROS2).
  • CameraInfo: se renombran campos D/K/R/P → d/k/r/p.
  • DiagnosticStatus.level: se ajusta al rango válido.

Un watchdog reinicia el puente si se queda "vivo pero sin datos" (el rosbridge del MiR se cuelga seguido). Hoy el puente está apagado: solo se necesita con el MiR activo.

8. El visor 3D

Reusa los mismos datos de joints, sin conexión nueva.

El visor no abre ninguna conexión propia: lee del mismo poller de /api/ur/joints (paso 3). Con esos 12 ángulos y la cadena cinemática del robot (ur5e_chain.json, sacada del modelo URDF), dibuja cada brazo con primitivos (cilindros para los eslabones, esferas para las articulaciones) usando Three.js, y rota cada articulación según su ángulo en vivo.

Se usan primitivos y no las mallas realistas .dae porque esas, aunque cargan, no se renderizan en el navegador (un problema del archivo). El modelo usa la orientación del URDF; recuerda que el montaje físico real va al revés.

9. Glosario rápido

TérminoQué es
ROS 2 topicUn "canal" de publicación/suscripción. Un nodo publica (ej. /joint_states) y otros escuchan. Flujo continuo.
ROS 2 actionPara tareas que tardan (ej. mover el brazo): mandas una meta y recibes el resultado al terminar. Lo usamos para mover joints.
ROS 2 servicePetición → respuesta inmediata (ej. set_payload). Como una función remota.
DDSEl "transporte" interno de ROS 2. Gracias a network_mode: host, descubre nodos entre contenedores sin configurar.
rosbridgeUn puente ROS ↔ WebSocket, para que un navegador (que no habla ROS) pueda suscribirse a topics. Lo usamos para la cámara.
RTDEEl protocolo propio de Universal Robots para hablar con los brazos en tiempo real (TCP).
RESTAPI por HTTP (peticiones GET/POST que devuelven JSON). Así habla el MiR y así habla nuestro backend con el navegador.
roslib.js / roslibpyLibrerías para hablar con rosbridge desde JavaScript (navegador) y desde Python (el puente del MiR).

10. Por qué está armado así (decisiones)

DecisiónMotivo
network_mode: hostEl kernel de la Jetson no soporta las redes bridge de Docker. Compartir la red del host hace que localhost y DDS funcionen entre contenedores.
rosbridge dentro de mir_ur_driverCuando estaba en un contenedor aparte había problemas de descubrimiento de topics. Junto al driver, son locales.
joint_server.py (HTTP) en vez de docker execUn servidor que se suscribe una vez es ~10× más rápido que abrir un proceso por cada lectura.
Workspace de Eemil montado (no copiado)Usamos sus paquetes (duo_ur, etc.) sin modificar su código. Regla de aislamiento del proyecto.
El driver UR no arranca soloEl usuario lo inicia desde la UI, para no intentar conectarse a brazos apagados o en error.

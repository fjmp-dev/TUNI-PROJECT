Paso 1: Leer los joints sin reiniciar nada

El driver UR de Eemil (duo_ur_real.launch.py) ya estaba corriendo en nuestro contenedor mir_ur_driver y publicando las posiciones de los 12 joints en el topic /joint_states a 400 Hz. El reto era hacer que esos datos llegaran a la pagina web.

Intentamos dos cosas que no funcionaron bien: conectarnos al rosbridge desde JavaScript (fallaba por un problema de calidad de servicio) y ejecutar un proceso nuevo por cada consulta (demasiado lento, ~200ms por peticion). La solucion fue crear joint_server.py, un mini-servidor HTTP que vive dentro del contenedor, se suscribe a los joints una sola vez y cada que la pagina pregunta, responde con los ultimos valores en JSON. Esto es 10 veces mas rapido y no gasta recursos.

Lo desplegamos sin reiniciar el contenedor (para no perder la conexion con los brazos). Usamos docker cp para copiar el archivo y docker exec -d para lanzarlo en segundo plano.

Paso 2: Mover los codos desde la pagina

Una vez que ya veiamos los joints, necesitabamos poder moverlos. Creamos joint_mover.py, un script que recibe tres parametros -- que brazo, que articulacion, y cuanto mover -- y se comunica con el controlador del robot para ejecutar la trayectoria.

En la pagina agregamos 8 botones organizados por brazo: paso grande (+/- 0.1 radianes, como 6 grados) y paso chico (+/- 0.01 radianes, como medio grado). Los botones llaman al backend, que ejecuta el script dentro del contenedor, y el brazo se mueve.

Por seguridad, solo movemos los codos. Las munecas no se tocan. Ademas cada boton tiene un delay para que no puedas enviar dos comandos al mismo tiempo.

ProblemaQue pasabaComo lo arreglamos
1. Programa de control detenido El controller reportaba "movimiento exitoso" pero el brazo fisico no se movia ni un milimetro. El programa URScript dentro del robot se habia detenido. Agregamos una llamada al servicio resend_robot_program que reenvia el script de control al robot. Ahora se hace automatico al iniciar el driver y tambien como recuperacion si un movimiento falla.
2. Controller apagado en brazo derecho El brazo derecho rechazaba todos los comandos con "goal rejected". El right_controller_stopper (un sistema de seguridad interno del UR driver) habia apagado el controlador. Agregamos una reactivacion automatica del controlador con ros2 control switch_controllers. El script de inicio lo intenta 3 veces con pausas de 3 segundos.
3. Scaled controller roto Habia un controlador "escalado" que supuestamente es mas seguro, pero estaba roto: aceptaba comandos pero jamas los ejecutaba. Se quedaba esperando respuesta para siempre. Volvimos al controlador estandar de ROS2 (joint_trajectory_controller), que funciona perfectamente. El escalado lo dejamos apagado.

El sistema de recuperacion automatica

Para que la pagina sea confiable, el backend detecta automaticamente cuando un movimiento falla. Si el error es "goal rejected", intenta recuperarse: reenvia el programa de control al robot, reactiva el controlador, espera un segundo, y reintenta el movimiento. Si funciona, la pagina muestra "OK (recovered)". Si no, muestra el error especifico para que sepas que hacer.

Lo que aprendimos

Sobre los robots

Los UR5e que tenemos usan PolyScopeX (el software nuevo de Universal Robots). Esto significa que no tienen el servidor de dashboard clasico (solo disponible en e-Series). Para recuperar al robot de un error, hay que hacerlo manualmente desde la tablet.

Sobre la arquitectura

Para datos en tiempo real, un servidor HTTP persistente dentro del contenedor es mucho mejor que ejecutar procesos nuevos por cada consulta. Es 10 veces mas rapido, no gasta recursos, y se puede desplegar sin reiniciar el contenedor.

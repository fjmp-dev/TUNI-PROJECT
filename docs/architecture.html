1. Resumen ejecutivo

El objetivo es controlar el robot MIR + dos UR5e desde un navegador, sin instalar ROS en el portátil del usuario. La solución actual corre en el Jetson AGX Orin ("Kevin", IP 192.168.1.75) y usa Docker + ROS 2 Humble + MoveIt + WebSocket (rosbridge) + una UI web.

Estado actual (crítico): el brazo derecho se mueve, pero los movimientos no son seguros. El right_joint_trajectory_controller falla repetidamente por violación de tolerancia, y el action_bridge forzaba una orientación fija que torció la muñeca. NO usar el robot físico hasta corregir el controlador y probar en simulación.

Relación con proyectos originales

mir_ur_driver no copia el workspace de UR; lo monta desde /home/lab/pbd_system. Esto significa que:

  • Usamos los paquetes duo_ur, moveit_utils_pkg, duo_ur5e_torso_moveit_config, etc., sin modificarlos.
  • Cualquier cambio en /home/lab/pbd_system afectaría a los proyectos originales, así que no lo tocamos.
  • Nuestros scripts (action_bridge.py, ur_entrypoint.sh) se montan dentro del contenedor desde mir_suite/scripts/.

5. Código y configuración clave

5.1 docker-compose.yml

Define los servicios. El servicio ur_driver monta nuestros scripts y config, y el workspace de pbd:

ur_driver:
  image: mir-ur-driver:latest
  container_name: mir_ur_driver
  network_mode: host
  privileged: true
  volumes:
    - /home/lab/pbd_system:/root/workspace
    - ./scripts/action_bridge.py:/action_bridge.py
    - ./scripts/ur_entrypoint.sh:/entrypoint.sh
    - ./config:/mir/config

5.2 scripts/ur_entrypoint.sh

Arranca todo dentro de mir_ur_driver:

  1. Lanza duo_ur_real.launch.py.
  2. Fix crítico: espera a que /move_group exista y fuerza use_sim_time:=false. El launch file original lo hardcodea a true, lo cual rompe MoveIt en hardware real.
  3. Lanza rosbridge en :9090.
  4. Lanza action_bridge.py.
  5. Activa left_joint_trajectory_controller y right_joint_trajectory_controller.

5.3 scripts/action_bridge.py

Recibe JSON por /mir/command y traduce a actions de ROS 2.

Comandos soportados

ComandoArgsAction resultante
move_armside, x, y, z, frame, velocity/move_action (right) o /move_to_pose_in_frame (left)
homeside, velocity/move_action con goal en joint-space a la pose segura guardada
stop_arm—Publica evento (no cancela goals aún)
close_handgrasp_type, max_currents/left_humanoid_hand/close_hand
open_hand—/left_humanoid_hand/open_hand

Para el brazo derecho, ahora obtiene la orientación actual del TCP vía TF y la mantiene en el goal. Antes forzaba Quaternion(0,0,0,1), lo que torcía la muñeca.

7. Simulación

Para dejar de probar con el robot físico, tenemos dos caminos:

7.1 use_fake_hardware:=true (más fácil)

El launch file lo soporta, pero tiene un bug. Si se corrige, arranca un mock de hardware que publica /joint_states y permite probar MoveIt sin robot.

Bug conocido: en duo_ur_real.launch.py línea ~511 se hace controllers_active.remove("tcp_pose_broadcaster"), pero los nombres reales son left_tcp_pose_broadcaster y right_tcp_pose_broadcaster. Hay que cambiar esa línea o crear un launch wrapper propio.

Comando de prueba (una vez corregido):

ros2 launch duo_ur duo_ur_real.launch.py \
  use_fake_hardware:=true \
  launch_rviz:=false \
  headless_mode:=true \
  controller_spawner_timeout:=60

7.2 URSim (más realista)

El simulador oficial de UR. Existen scripts en el workspace:

/home/lab/Desktop/MIR/pbd_system/ros_ws/src/official_repos/Universal_Robots_ROS2_Driver/ur_robot_driver/scripts/start_ursim.sh

Requiere levantar URSim (Docker/VM) y apuntar el driver a la IP del simulador. Es más fiel pero también más trabajo.

7.3 Recomendación

Usar use_fake_hardware:=true corregido para validar la arquitectura, el action_bridge y la UI. Luego pasar a URSim para validar trayectorias antes de volver al robot físico.

8. Roadmap propuesto

  1. Simulación segura
    • Corregir/Crear launch para use_fake_hardware:=true.
    • Probar action_bridge y UI contra simulación.
  2. Controlador correcto
    • Cambiar MoveIt a right_scaled_joint_trajectory_controller.
    • Validar en simulación y luego en robot real con movimientos muy pequeños.
  3. Seguridad en la UI
    • Botón Home con confirmación.
    • Límite de velocidad máxima (ej. 5%).
    • Indicador de estado de conexión de cada brazo.
    • Botón de parada de emergencia que cancele goals reales.
  4. Brazo izquierdo
    • Diagnóstico de hardware (teach pendant, URCap, red).
  5. Integración MiR + mano + Nordbo
    • Panel de control del base MiR.
    • Panel de la mano (open/close).
    • Visualización del sensor F/T.

El MiR200 es un robot movil que navega por el laboratorio. Tiene su propio sistema ROS1 interno con 199 topics de telemetria: posicion, velocidad, escaneres laser, mapa, bateria, estado del robot, etc. Nuestro objetivo era traer esos datos a nuestro sistema ROS2 en Kevin y mostrarlos en la pagina web.

Trabajamos con los scripts que Eemil ya habia creado (mir_raw.py y mirror_node.py) sin modificarlos. Creamos un contenedor Docker dedicado para el bridge y un sistema de recuperacion automatica para cuando el MiR se cuelga.

Lo que construimos

Creamos un contenedor Docker llamado mir_mir que corre el mir_raw.py de Eemil (sin modificarlo, solo lo montamos como volumen). Este script se conecta al rosbridge del MiR en el puerto 9090, descubre automaticamente los 199 topics, y crea publicadores ROS2 para cada uno. Asi, los datos del MiR aparecen como topics nativos de ROS2 en Kevin.

Ademas, construimos un proxy REST en el backend (FastAPI) que consulta la API del MiR cada pocos segundos y devuelve los datos en un JSON limpio para la pagina web. La API REST es mas estable que el rosbridge y funciona incluso cuando el robot esta en Pause.

El sistema de auto-recuperacion

Problema: El MiR200 es inestable. Su rosbridge se cuelga con frecuencia (especialmente cuando el robot esta en modo Running). Cuando esto pasa, el mir_raw.py se queda "vivo pero sin datos": el proceso sigue corriendo pero nunca recibe mas mensajes. No se muere, no se reconecta, solo se queda congelado.

Solucion: Construimos un watchdog -- un script que revisa cada 30 segundos si el bridge esta realmente procesando datos. Si pasan 90 segundos sin actividad, el watchdog mata el proceso. El entrypoint del contenedor tiene un ciclo infinito que lo relanza automaticamente. Resultado: cuando el MiR se recupera, el bridge se reconecta solo, sin intervencion humana.

El modulo del MiR en la pagina web

Construimos un panel visual en la pagina web que muestra la informacion del MiR200 usando su API REST (puerto 80). Esta API funciona incluso cuando el robot esta en Pause, lo cual es una gran ventaja porque el rosbridge se cae en ese estado.

Conclusion: El robot solo es estable en Pause. Al darle Play, los servicios de red colapsan. Esto es causado por los errores 9000 (sistema de seguridad), no por nuestro software.

Recomendacion

  1. Reinicio fisico completo del robot (Shutdown desde tablet, esperar 1 minuto, encender)
  2. Si los errores 9000 persisten: revisar cables del PLC SICK y boton de emergencia
  3. Contactar soporte MiR si el hardware esta danado
